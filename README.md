# Problem 021 — 校园管理系统 (Campus Management System)

## 1. 项目概述

你需要**从零开始**构建一套 **校园管理系统**，包含后端 API 服务和前端 Web 界面。
系统主要面向三种用户角色：**管理员 (Admin)**、**教师 (Teacher)** 和 **学生 (Student)**。
> 需要保证系统可以通过简单的命令在本地启动和运行。

---

## 2. 功能需求

### 2.1 用户与认证

| 编号 | 功能 | 描述 |
|------|------|------|
| AUTH-01 | 登录 | 所有角色通过用户名 + 密码登录，返回 JWT 或等效 Token |
| AUTH-02 | 角色鉴权 | 不同 API 端点需要不同角色权限；越权访问应返回 403 |
| AUTH-03 | 个人信息 | 已登录用户可查看和修改自己的基本信息（姓名、邮箱等） |

### 2.2 管理员功能

| 编号 | 功能 | 描述 |
|------|------|------|
| ADM-01 | 教师管理 | 增删改查教师（工号、姓名、院系、邮箱） |
| ADM-02 | 学生管理 | 增删改查学生（学号、姓名、班级、年级、邮箱） |
| ADM-03 | 课程管理 | 增删改查课程（课程号、名称、学分、描述） |
| ADM-04 | 开课 (Class) 管理 | 创建 / 修改某门课程在某学期的开课记录，并指定授课教师、上课时间 |
| ADM-05 | Dashboard | 展示学生总数、教师总数、课程总数、本学期开课数等统计 |

### 2.3 教师功能

| 编号 | 功能 | 描述 |
|------|------|------|
| TCH-01 | 查看所授课程 | 列出本人教授的所有开课记录 |
| TCH-02 | 查看选课学生 | 查看某一开课下的已选课学生名单 |
| TCH-03 | 成绩录入 | 为某一开课的每位学生录入分数（0-100） |
| TCH-04 | 成绩修改 | 可修改已录入的成绩 |

### 2.4 学生功能

| 编号 | 功能 | 描述 |
|------|------|------|
| STU-01 | 选课 | 浏览可选开课并选课（不超过学分上限：默认 30 学分/学期） |
| STU-02 | 退课 | 在成绩录入前可以退课 |
| STU-03 | 查看课表 | 以时间表形式展示本学期已选课程 |
| STU-04 | 查看成绩 | 查看自己各门课的成绩 |

---

## 3. 系统架构要求

### 3.1 后端

- 提供 RESTful API（或 GraphQL，自行选择）
- 基础的输入校验和错误处理，返回统一的 JSON 格式
- 数据库存储（关系型推荐，如 SQLite / PostgreSQL / MySQL）
- 强制技术栈：
  - Java + Spring Boot

### 3.2 前端

- 单页应用 (SPA)
- 响应式布局，移动端友好
- 至少包含以下页面/视图：
  - 登录页
  - 管理员 Dashboard
  - 学生 / 教师 / 课程 / 开课的 CRUD 列表与表单
  - 选课 / 退课界面
  - 成绩录入 / 查看界面
  - 个人课表
- 强制技术栈：
  - Vue 3 + Element Plus / Vuetify

### 3.3 数据库

概念数据模型（字段仅供参考，可自行扩展）：

```text
User        (id, username, password_hash, role, name, email, created_at)
Teacher     (id, user_id→User, staff_no, department)
Student     (id, user_id→User, student_no, grade, class_name)
Course      (id, code, name, credits, description)
Class       (id, course_id→Course, teacher_id→Teacher, semester, schedule, capacity)
Enrollment  (id, class_id→Class, student_id→Student, enrolled_at)
Grade       (id, enrollment_id→Enrollment, score, updated_at)
```

## 5. 测试与验收标准

### 5.1 冒烟测试（自动）

`/workspace/data/21/tests/smoke_test.sh` 是一个自动化冒烟测试脚本。
启动服务后执行该脚本，它会依次：

1. 以管理员身份登录，获取 Token
2. 创建一名教师
3. 创建一名学生
4. 创建一门课程
5. 创建一个开课记录
6. 以学生身份选课
7. 以教师身份录入成绩
8. 以学生身份查看课表
9. 以学生身份查看成绩

每一步会对比实际响应与 `expected_responses/` 目录中的预期结果，输出 PASS / FAIL。

> **通过标准**：冒烟测试 9 个步骤全部 PASS。

### 5.2 手动验收（自评）

除了冒烟测试，你还应当进行以下自评：

- [ ] 前端页面在浏览器中可正常访问
- [ ] 管理员可以完成教师 / 学生 / 课程 / 开课的增删改查
- [ ] 教师可以查看自己的课程和学生名单，录入 / 修改成绩
- [ ] 学生可以选课 / 退课、查看课表和成绩
- [ ] 非法角色访问受限 API 返回 403
- [ ] 数据库数据持久化（重启后端后数据不丢失）

## 8. 进阶与扩展要求（建议实现）

以下内容用于进一步提升题目难度和系统完备度，不要求一次性全部完成，但鼓励尽量覆盖：

### 8.1 权限模型与审计日志

- 引入更细粒度的权限控制（例如基于角色+资源+操作的 RBAC 模型）
- 对关键操作记录审计日志，例如：
  - 管理员创建 / 删除用户、课程、开课
  - 教师修改成绩
  - 学生选课 / 退课
- 审计日志至少需要包括：操作者 ID、时间戳、操作类型、目标实体、变更前后摘要（可序列化为 JSON）。

### 8.2 多学期与多学院支持

- 支持多个学期（semester）并行存在，查询接口必须支持按学期过滤：
  - 学生课表：默认显示当前学期，可切换历史学期
  - 成绩查询：可按学期汇总 GPA / 学分
- 支持多学院 / 多专业：
  - 教师和课程归属到学院
  - 管理员 Dashboard 中增加按学院的统计视图

### 8.3 并发与数据一致性

- 设计并实现**选课并发控制策略**，避免以下情况：
  - 课程容量为 30，却因并发选课导致选中 31 人以上
  - 重复选课（同一学生对同一开课多次提交）
- 可选实现方式：
  - 数据库事务 + 行级锁 / 悲观锁 / 乐观锁（版本号字段）
  - 对关键写操作加去重约束（唯一索引：`(class_id, student_id)`）
- 在 `README_log.md` 中说明你的设计选择和可能的边界问题。

### 8.4 非功能性要求（性能 / 安全 / 可维护性）

- **性能**
  - 对常用查询添加合理索引（如按学号、课程号、学期的组合索引）
  - 支持分页查询，避免一次性返回过大数据集
- **安全**
  - 密码必须加盐哈希存储，不允许明文
  - 对所有写操作进行基础的输入校验和权限检查
  - 防止常见注入问题（SQL 注入、XSS 等）
- **可维护性**
  - 后端代码应按“领域模块”（用户 / 课程 / 选课 / 成绩）组织，而非巨型单文件
  - 前端组件按页面 / 业务模块拆分，避免过度耦合

### 8.5 DevOps 与运维（进阶）

- 提供简单的容器化部署方案（如 Dockerfile + docker-compose.yml），实现：
  - 一条命令启动数据库和应用（前后端可视为单服务或拆分）
  - 支持通过环境变量配置数据库连接、JWT 密钥等敏感信息
- 日志输出标准化：
  - 后端使用统一格式（例如 JSON 行），包含时间、级别、请求 ID 等字段
  - 针对错误日志，记录请求路径、用户 ID（如有）、堆栈摘要
